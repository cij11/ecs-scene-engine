## Title
task-ESE-0006: Build agile service

## Description
Replace the scattered automation scripts with a cohesive agile service. Three-layer architecture:

- **Controller** — CLI entry point (`npm run agile -- <command>`). Input validation, top-level error handling.
- **Service** — Business logic. Expects clean input. Enforces gates, status transitions, velocity calculation.
- **Repository** — Persistence via JSON file read/write.

### Data model

Each ticket is a single JSON file with a UUID (`id` field). UUIDs are generated by the service and are immutable.

Rich text fields (description, acceptance criteria, comments) are markdown strings within JSON.

### Files

- `tickets/<uuid>.json` — one file per ticket. Contains all ticket state: id, title, status, description, AC, size, timestamps, etc.
- `ticket_id_relationships.json` — maps ticket filename/name (e.g. `feat-ESE-0005`) to UUID for easy reference.
- `statuses.json` — defines the valid status flow, allowed transitions, and gates per status.
- `sprints.json` — all sprint data: name, status, ticket UUIDs, estimates, actuals.
- `velocity.json` — velocity history across sprints.

### CLI commands

`npm run agile -- <command> [args]`

Ticket commands:
- `ticket create <type> <title>` — create a ticket with UUID
- `ticket status <name> <status>` — transition with gates
- `ticket accept <name>` — set demoAccepted (human only)
- `ticket validate` — check all tickets for consistency
- `ticket list` — list tickets with status

Sprint commands:
- `sprint create <name>` — create a sprint
- `sprint add <ticket> <sprint>` — add ticket to sprint
- `sprint start <sprint>` — start sprint, record estimates
- `sprint complete <sprint>` — complete sprint with gates
- `sprint velocity` — velocity report

### What this replaces

All scripts in `process/agile/automation/`. The entire `automation/` directory is replaced by the agile service in `tooling/agile-service/`.

## Acceptance Criteria
- Single CLI entry point: `npm run agile -- <command>`
- Controller validates input, service enforces business logic, repository handles persistence
- Each ticket is a JSON file with an immutable UUID
- ticket_id_relationships.json maps human-readable names to UUIDs
- Rich text fields (description, AC, etc.) are markdown strings in JSON
- All gates from current automation are preserved (readyForDev, inTesting, done, demo validation)
- Status transitions logged to audit trail
- Velocity correctly calculated from ticket timestamps
- Sprint completion reads status from ticket JSON (no markdown parsing)
- All existing npm sprint/ticket commands work through the new agile service
- Current automation scripts are removed after migration
